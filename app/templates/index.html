For Creators

AI video generation gets messy fast. Prompts drift, characters change, props disappear, and every shot can become a new negotiation with the machine.

RenderScript gives your screenplay scene a shooting pack before generation begins.

It helps you:

  • see the scene as shot cards
  • copy base prompts into your preferred AI video tool
  • organise references in scene folders
  • generate test takes
  • pick keepers
  • keep audio and post notes together

Workflow:

Write the scene.
Build the RenderPackage.
Copy a shot prompt.
Generate takes.
Pick keepers.
Cut the sequence.
Finish audio in post.

RenderScript does not replace direction or editing. It gives creators a clearer way to move from screenplay scene to AI-video rough cut.

About RenderScript

RenderScript prepares screenplay scenes for AI video production.

It converts one scene into a storyboard-led RenderPackage: shot cards, base prompts, reference folders, keeper tracking, post-production notes, and developer files for downstream integrations.

RenderScript is a universal AI-video workflow layer, not a platform-specific generator.

RenderScript does not generate video. It builds the shooting pack you use with your preferred AI video tools.

For Developers & Agents

RenderPackage now includes an agent-facing contract for downstream workflow automation. Agents consume the package after RenderScript builds it; they do not recreate the package and they do not execute providers unless a separate adapter and user credentials are supplied.

Creators use the Studio UI and the creator-facing package files. Developers use the CLI, GitHub workflows, and DEVELOPER_FILES when they need validation, adapters, orchestration, or pipeline integration.

DEVELOPER_FILES contains rpack.json, provenance.json, shot_list.csv, bindings.csv, AGENT_ORCHESTRATION.md, provider_capabilities.example.json, package_map.md, and provider prompt packs.

Agent contract

  • AGENT_ORCHESTRATION.md explains how local or external agents can consume the package.
  • rpack.json exposes shot IDs, shot intent, base prompts, reference requirements, reference folders, prompt pack paths, and provenance paths.
  • provider_capabilities.example.json is a template for adapter capability maps.

What developers can build

  • Provider adapters for specific AI video workflows
  • Validation and QA checks around screenplay and package outputs
  • Internal tools that wrap RenderPackage generation
  • Pipeline integrations for studio or cloud workflows
  • Downstream orchestration systems built around RenderPackage

Provider execution, account access, billing, media upload, render queues, and automatic keeper choice remain outside the MVP.

The machine-readable package contract lives in:

DEVELOPER_FILES/rpack.json
